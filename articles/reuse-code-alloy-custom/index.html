Reuse code in Alloy: a custom solution

Code reutilization is a key aspect of every project, helping developers keeping their code tidy and well tested. With this article I want to share a technique I often use in my Titanium Alloy projects for reusing components, libraries, widgets and assets in different mobile applications.

Alloy’s built-in solutions

Alloy itself is meant to help developers implement best practices in project source management. By building an MVC layer over a classical Titanium project and with its extensive use of CommonJS’ module pattern, it drives developer to code reutilization.
Built-in solution to reuse code and keep development D.R.Y. are CommonJS modules and Widgets.

CommonJS modules

CommonJS is the standard de facto for dependency injection in Javascript. It encapsulates functionalities, keeping code tidy and preventing global scope flooding issues. More over, package managers like npm make managing references extremely easy, thus you can both reduce coupling and increase cohesion in your source. Here you can learn how to create and reference CommonJS modules in Alloy. It’s also plenty of third party modules on gitTio.

Widgets

Widgets are meant to be reusable UI components. They’re basically an atomic Alloy MVC entity - a view, a controller, a stylesheet, optionally a model. They can grow in complexity by including other modules, controllers, views, libraries and widgets, but they often target simple solutions as they would become hard to costomize for the current project. Here’s the official, documentation.

A custom approach

As I work on a daily basis with Alloy, I found several limits on built-in aproaches. For example, CommonJS modules don’t allow you to use the nice MVC paradigm we all love, and the only way to include UI in a module would be using ol’ style Titanium code. If you have a fancy dialog you want to reuse then widgets are fine, but what if you have a set of two or three components you want to interact?

I wanted something that allows me to:

  • write my app the style I prefer, not being forced to adhere to an overkilling framework’s standard;
  • cherry-pick features in my project from different sources;
  • Don’t Repeat Myself, writing again the same routine nor copying&pasting code through projects;
  • focus on the app I’m developing instead of the environment I’m building (that means: automate repetitive tasks).

What if I can reference some other projects’ repositories, define the files I want to use in my app and the ones I don’t, and keep them updated automatically? Well, that’s actually what I already do!

So let’s dive into it.

Requirements

As you develop in Alloy, you’re already aware of almost everything we need: npm, Git, Titanium CLI - our bread-n-butter. The very last thing is Grunt, which if you don’t know is the state-of-the-art of task automation for javascript development. You may want to learn more about Grunt, but for the moment you are fine by executing this in your shell:

!#markdown
$ npm install -g grunt-cli

This may need sudo on certain systems, please refer to docs for any doubt.

Now create an Alloy project (or pick an existing one), then create two files in the root folder, and name them Gruntfile.js and package.json. Leave the first empty for the moment, and in the second simply put:

!#javascript
{	
  "name": "MyProject",
  "version": "0.0.0",
  "description": "",
  "main": "Gruntfile.js",
  "dependencies": {},
  "devDependencies": {
	 "grunt": "~0.4.5",
	 "grunt-git": "https://github.com/balanza/grunt-git/tarball/master"
  },
  "author": "",
  "license": "ISC"
}

just run npm install in the shell for installing dependencies.

Addressing submodules

Now we have to reference submodules. A Git submodules is, roughly, a repository cloned in a subfolder of your project and bound to the project’s repository. Is actually slighty more than that, but for the moment that’s all we need. Assuming your project is already a git repository, you can add a submodule in two ways:

  1. using a Git Client. For example, in SourceTree, go to Repository > Add Submodule... and specify the url of the repository you want to reference and the subfolder in which you want to clone the submodule. Don’t forget the branch (which is likely to be master, in case you don’t know why you need to select a branch).

  2. using command line, just exec

     !#markdown
 	$ git submodule add <git@ ... > ./submodules/commonFiles
 	$ cd ./submodules/commonFiles
 	$ git checkout master

From now on, every time we update our local project repository, submodule’s last changes will be downloaded into the ./submodules/commonFiles folder.

Adding automation

The following paragraph will let us build a grunt task that:

  1. update submodules
  2. cherry-pick files from submodules and copy them into our Alloy’s /app folder, making them available in our project.

You need to edit your Gruntfile. The following is trivial example I wrote. For the sake of simplicity there’s only one submodule, but they can obviously be more. Here’s the file you can copy&paste:

!#javascript
/**
 * Gruntfile.js
 *
 * This file collects automated tasks for this Alloy app.
 * Please refer to Grunt for documentation (http://gruntjs.com/)
 *
 * @author Emanuele De Cupis (http://emanuele.decup.is)
 */
module.exports = function(grunt) {

	// you'll need these modules for correctly addressing folders and files
	var path = require('path');
	var fs = require('fs');

	//root for our submodules
	var submodules = path.join(__dirname, 'submodules');
	var repoA = path.join(submodules, 'commonFiles');


	grunt.initConfig({

    	gitcheckout: {
        	repoA: {
            	options: {
                	cwd: repoA,
                	verbose: true,
               		branch: '.'
            	}
        	}
    	},
    	gitpull: {
        	repoA: {
            	options: {
                	cwd: repoA,
                	verbose: true,
                	callback: function(done) {
             			mergeFilesFromCommonFiles();
                   		done();
               		}
       			}
        	}
    	}
    	
	});


	/**
	 * init grunt plugins
	 */

	grunt.loadNpmTasks('grunt-git');


	/**
	 * register tasks
	 */

	//this task fetches last changes from submodule RepoA's remote master branch, then copy files
	// use 'grunt update-repoa' to run it in terminal
	grunt.registerTask('update-commonFiles', 'update commonFiles submodule and copy files in my app', ['gitcheckout', 'gitpull']);



	/**
	 * This function merges all files from submodule RepoA to current project RepoB
	 * Here you can implement your custom merge business logic, choosing which files to copy or not.
	 *
	 */
	function mergeFilesFromCommonFiles() {
	
    	/**
     	 *  Our simple merge business logic simply copies all files from following folders into our project
     	 */

    	var toBeCopied = ['controllers', 'styles', 'views', 'lib'];

    	for (var i = 0; i < toBeCopied.length; i++) {
        	var folder = toBeCopied[i];
            var dir = path.join(repoA, 'app', folder);
	        var files = fs.readdirSync(dir);
            for (var j = files.length - 1; j >= 0; j--) {

            	var file = path.join(dir, files[j]);
            	var source = file;
            	var destination = path.join('./app', folder, files[j]);
            	copyFile(source, destination);
        	}
    	};

	    console.log('commonFiles copied!');
    }	  
    
    /**
     * Simple routine to copy files in js
     * @param  {string} srcFile   path for source file
     * @param  {string} destFile  path for destination file
     * @return {object}           reference to the created file
     */
    function copyFile (srcFile, destFile) {
        console.log('copying ' + srcFile + ' into ' + destFile);
        var BUF_LENGTH, buff, bytesRead, fdr, fdw, pos;
        BUF_LENGTH = 64 * 1024;
        buff = ''; // new Buffer(BUF_LENGTH);
        fdr = fs.readFileSync(srcFile);
        fdw = fs.writeFileSync(destFile, fdr);
        return fdw;
    };  	    

};

If you are not expert of Grunt it may appear complex. It’s not my purpose to dive into Grunt’s mechanisms, but let me explain what’s going on:

  • in the grunt.initConfig function we pass the definition for the Git commands we want to execute on our submodule. The submodules is addressed by the variable repoA which holds the path to the submodule relative to our project. The gitpull and gitcheckout commands are available thanks to the Grunt plugin grunt-git (which we already installed togheter with all our npm dependencies at the beginning of this article). For more information about, check out the plugin’s documentation.
  • the mergeFilesFromCommonFiles function implements our cherry-pick mechanism. Better said, here we choose the files we want from ./submodules/commonFiles and copy them into ./app. In this example we are assuming the submodule is an Alloy app itself, and we are copying controllers, views, styles and lib folders.
  • the grunt.registerTask function defines a task named update-commonFiles as a combination of gitcheckout and gitpull commands.

Usage

As the task is defined and registered in the Gruntfile, we can perform submodules update every time by executing the following command in the root of our project:

!#markdown
$ grunt update-commonFiles

As we defined, it automatically checkout the master branch, pull from remote repository and copy files for using them in our app.

Example

I’ve shared a barebone example on my Github page. I’ve created an Alloy app named RepoB which make use of components from another app, RepoA. Those apps do actually nothing, they just serve as an example for what we were discussing in this article. To run the example, execute the following commands:

!#markdown
$ git clone git@github.com:balanza/reuse-code-alloy-repoB.git 
$ cd reuse-code-alloy-repoB
$ npm install 
$ grunt update-repoa
